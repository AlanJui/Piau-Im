我在 mod_ca_ji_tian.py （查字典）模組，其中有一函數： han_ji_ca_piau() ，其功能為：依據我傳入的【漢字】、欲查得【讀音類別】（文讀音/白話音），在【漢字庫】資料庫 （SQLite）查尋，並回傳查得之漢字【讀音】（以【台語音標】格式標示）。

每個漢字的【讀音】，以【台羅拼音】格式標注，存在【漢字庫】資料庫，其中的【漢字庫】資料表（Table）中。在【漢字庫】資料表，每筆【資料紀錄】（Record），有個【常用度】欄位。此欄位目前可用來識別，該筆【資料紀錄】是【文讀音】或【白話音】，其區別規則如下：

- 文白通用音：1 <= 常用度 < 0.8
- 文讀音：0.8 <= 常用度 < 0.6
- 白話音：0.6 <= 常用度 < 0.4
- 方音/罕見音：0.4 <= 常用度 < 0

目前此函數，能如預期正常運作。但有個問題，就是函數查得之讀音，往往都不是我所要的。最常見的情況是：

同一漢字，有不同的方音腔：在泉州腔、漳州腔、廈門腔，有不同的讀音，但其【常用度】值都是 0.8 （表【文讀音】），或都是 0.6 （【白話音】）；

同一漢字，當【名詞】、【動詞】、【形容詞】時，各有不同【讀音】：但同樣如上述一般，其【常用度】值亦皆相同。

對此現象，不知您是否有什麼演算法可解此問題？若是暫無，我希望能幫我變更目錄的 query 方式：

1. 第一優先：以【漢字】、【常用度】三欄資料，篩選符合的【資料紀錄】；
1. 第二優先：再依【更新時間】欄排序，令【與今天此刻最近的】的【資料紀錄】排在最前面。

【註】：因為現在我習慣在每一個 Excel 檔完成【漢字標音】後，利用 a330.py 程式，取用【標音字庫】工作表的每個登錄的漢字標音資料紀錄，更新資料庫中【漢字庫】資料表紀錄的【更新時間】欄。

我看了 han_ji_ca_piau_im() 的實作與 漢字庫 資料表結構，以下是我的看法。

問題本質：常用度欄位「超載」了
您遇到的不是演算法問題，而是資料模型問題。【常用度】一個欄位同時承擔了兩種語意：讀音類別（文/白/其它）加上優先順序。但「方音腔別」（泉/漳/廈）與「詞性」（名/動/形）這兩個維度，資料庫裡根本沒有存。查詢演算法再怎麼聰明，也無法從不存在的資料中變出區別——多筆紀錄常用度同為 0.8 時，資料庫層面它們就是無法區分的。

治本的做法有兩條路，供您日後參考：

加欄位：新增【腔口】、【詞性】欄位，查詢時依使用者偏好（如「漳州腔優先」）排序。這是最直接的解法，但需要回頭補資料。
依上下文選音：詞性歧義理論上要靠「斷詞＋詞庫」解決（先比對多字詞，如「行踏」的「行」與「銀行」的「行」查不同音）。這才是真正的演算法解，但工程量大，屬於長期方向。
對您提議的 query 變更：可行，且設計得不錯
您的方案本質上是把 a330 的人工校正流程變成一個「回饋訊號」——最近被人工確認過的讀音優先。這是很務實的輕量解法，我支持。具體看法：

排序改為 ORDER BY COALESCE(常用度,0) DESC, 更新時間 DESC 即可。因為 更新時間 格式是 YYYY-MM-DD HH:MM:SS，文字排序等同時間排序，不需轉型。同一常用度的多筆紀錄，最近被 a330 更新過的自然排最前。fallback 查詢（查無符合類別時取全部讀音那段）也應套用同樣排序。
以【漢字＋台羅音標＋常用度】三欄篩選：這只適用於「呼叫端已知音標」的場景（例如從標音字庫回填）。一般查音時呼叫端不知道音標，所以這應該做成選用參數（tai_lo_im_piau=None 時不加此條件），而非改變函數原本的介面語意。
兩個需留意的副作用
第一，更新時間 的語意會被污染。它原本是稽核欄位（「這筆資料何時被改過」），現在兼任「優先權」。若日後您因其他原因改了某筆紀錄（如修摘要說明），它的優先權也會被意外拉高。更乾淨的做法是另加一個【最近揀用時間】或【揀用次數】欄位專供排序，讓 a330 更新那個欄位，更新時間 維持原義。

第二，「最近用的優先」是無上下文的。上一份文件把某字當動詞校正過，下一份文件若該字是名詞，仍會先查到動詞音，您還是得人工改。若您的文件類型、腔口偏好穩定（看來是如此），這個機制會越用越準；若文件屬性多變，命中率就不穩定。

順帶一提：若走這方案，建議在資料庫加複合索引 (漢字, 常用度, 更新時間)，查詢會更快，尤其批次標音時。

總結：您的方案我認為值得做，建議微調為「時間欄位另立、做成次要排序鍵、音標篩選做成選用參數」。等您決定後，我可以再幫您實作。
