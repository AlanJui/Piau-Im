# 羅馬拼音字母置入聲調符號指引 (Tone Mark Placement Guide)

## 版本：V1.0

本指引旨在規範閩南話羅馬拼音（白話字、台羅、閩拼）在數位環境下的聲調符號標準、Unicode 編碼選用及字型呈現最佳實務。

---

## 1. 白話字 (POJ) / 台羅 (Tâi-lô) / 台語音標 (TLPA)
這三者在傳統音韻學的「八聲調」編號上是一致的。

| 調號 | 調名 | 調符名稱 | Unicode 編碼 | 範例 (a) | 備註 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | 陰平 | (無) | (不適用) | a | |
| 2 | 陰上 | 銳音符 (Acute) | `\u0301` | á | |
| 3 | 陰去 | 抑音符 (Grave) | `\u0300` | à | |
| 4 | 陰入 | (無/促音) | (不適用) | ak | 結尾為 p, t, k, h |
| 5 | 陽平 | 揚抑符 (Circumflex) | `\u0302` | â | |
| 6 | 陽上 | 倒折號 (Hacek) | `\u030C` | ǎ | 台羅專用 (部分方言) |
| 7 | 陽去 | 橫線 (Macron) | `\u0304` | ā | |
| 8 | 陽入 | 直線 (Vertical Line) | `\u030D` | a̍k | 組合垂直線，置於字母上方 |

## 2. 閩拼方案 (Bân-phing, BP)
閩拼方案的調號編排順序與前述系統不同，其設計邏輯更貼近現代漢語拼音的視覺習慣。

| 閩拼調號 | 對應八聲 | 調符名稱 | Unicode 編碼 | 範例 (a) | 備註 |

| 1 | 陰平 (1) | 橫線 (Macron) | `\u0304` | ā | **注意：與台羅不同，1聲有調符** |
| 5 | 陽平 (5) | 銳音符 (Acute) | `\u0301` | á | |
| 2 | 陰上 (2) | 倒折號 (Hacek) | `\u030C` | ǎ | |
| 6 | 陽上 (6) | 倒折號 (Hacek) | `\u030C` | ǎ | |
| 3 | 陰去 (3) | 抑音符 (Grave) | `\u0300` | à | |

## 3. 字型選用與技術避坑要點

### I. 字型選用 (Font Selection)
*   **首選字型**：**Charis SIL**。
*   **理由**：這是目前語言學界處理「組合字元 (Combining Diacritics)」最完美的字型。它完整支援 OpenType 的 `mark` (標記定位) 與 `mkmk` (標記對標記定位) 特性。
*   **備選**：Gentium Plus, Noto Sans TC (僅供漢字參考)。

### II. 避坑要點 1：組合字元 (Combining) vs. 預組字元 (Precomposed)
*   **避坑做法**：程式處理標音時，一律使用「組合字元」（如字母 `a` + `\u030D`）。
*   **理由**：預組字元（一個編碼代表帶調字母）在 Unicode 中不齊全（例如找不到帶垂直線的 `e`），會導致同一行標音中，有的字母是預組，有的字母是組合，造成視覺上的字體不統一。

### III. 避坑要點 2：零聲母位移 (The Anchor Problem)
*   **現象**：當標音以母音開頭（如 `e̍k`）且位於 HTML 標籤的第一個字元時，第 8 聲的垂直線常會向左偏離。
*   **解決方案**：在標音字串的最前端強制插入一個 **`\u200B` (零寬度空格)**。
*   **原理**：這能提供一個隱形的「字元錨點」，讓瀏覽器正確計算組合調符的中心位置。

### IV. 避坑要點 3：CSS 渲染特性
*   **必要設定**：在 CSS 中必須針對拼音類別加入以下宣告：
    ```css
    font-variant-ligatures: common-ligatures;
    font-feature-settings: "ccmp" 1, "mark" 1, "mkmk" 1;
    ```
*   **理由**：這能強制啟動字型內部的 OpenType 引擎，自動處理複雜的符號對齊與字形變換（如 POJ 的 `o͘`）。

### V. 避坑要點 4：正規化 (Normalization)
*   **做法**：在顯示或存儲轉換結果前，執行 `text.normalize("NFD")`。
*   **理由**：NFD (Decomposed) 格式將字母與調符拆分為獨立編碼，最能保證字型渲染引擎（如 Charis SIL）觸發精確的標記定位規則。
